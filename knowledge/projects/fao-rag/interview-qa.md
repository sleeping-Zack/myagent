# 法奥机器人智能客服平台面试问答

## Q1：这个项目是实习还是校企合作？你的角色是什么？

**参考回答：**

这是南昌大学与法奥机器人的校企合作项目，不是正式实习，没有劳动合同关系。我的角色是 Agent 后端开发。

项目周期 2026.01—2026.06，我参与基于 AIFlowy 平台的二次开发，重点负责 Direct/Agentic 路由改造、动态规划模块、QA 数据闭环和售后工单功能，以及 SSE 在 Nginx 下的长连接稳定性配置。

## Q2：AIFlowy 是什么？你不是在重复造轮子吗？

**参考回答：**

AIFlowy 是一个已有 RAG、多模型接入、插件系统、会话管理等基础能力的大型 AI 应用平台。我们不是从零开发整个平台，而是基于它进行功能扩展和业务改造。

我的工作是：
1. 增加面向机器人客服场景的业务逻辑（意图路由、动态规划、QA 沉淀）
2. 改造和优化现有执行链路（从单一 Agent 模式增加 Direct 快速通道）
3. 解决平台在客服场景下的具体工程问题（SSE 稳定性、幂等写入等）

平台提供了基础能力，我负责业务场景落地，这是企业 AI 应用开发的常见模式。

## Q3：Direct / Agentic 路由是怎么实现的？

**参考回答：**

原有链路对所有请求都走完整 Agent 流程（规划 → 工具执行 → 生成），资源消耗和延迟都偏高，简单问题体验差。

改造方案是在入口加一个意图判断层：
1. 规则层：用关键词和模式匹配快速识别明确的简单对话（问候、感谢、确认等）
2. 模型层：规则未命中的情况，用 Qwen 做意图分类，判断是简单对话还是需要工具的复杂请求
3. 路由决策：简单对话直接调用对话模型返回；复杂请求进入知识库、Workflow、插件和 MCP 链路

---

## Q4：动态规划模块具体做了什么？

**参考回答：**

Bot 在处理复杂任务时需要先生成执行计划再逐步执行。动态规划模块负责：

1. 根据当前问题和 Bot 已绑定的工具列表，生成 3—5 步的执行计划
2. 校验每步的工具类型、工具 ID 和参数是否合法
3. 异常时降级（跳过无效步骤或回退到直接回答）
4. 通过 SSE 实时推送计划生成状态、每步执行进度和最终完成状态

这样用户在等待期间能看到 Agent 在做什么，而不是只看到一个转圈。

## Q5：QA 数据闭环是什么？为什么要做这个？

**参考回答：**

客服系统产生大量真实对话，这些对话是很有价值的知识资产，但如果只是存在数据库里不利用，就浪费了。

QA 数据闭环的思路是把高质量的历史对话提炼出来，回流到知识库：

1. 从历史会话中抽取 User—Assistant 问答对
2. 消息配对和去重（以 `message_id` 做幂等去重）
3. 过滤低质量条目（太短、无意义的内容）
4. 人工审核界面：标注、审批、打标签
5. 审核通过的 QA 对支持批量 Markdown 导出
6. 导出内容可以直接导入知识库

GitHub 最新提交可以确认问答精选管理、审核、标签、导出 Markdown 和知识库导入功能，可查阅代码。

## Q7：如何快速进入一个陌生的大型 Java 项目？

**参考回答：**

我在 farino 项目中的实践是：

1. 先找启动入口（Spring Boot 主类）和核心配置，建立全局视图
2. 从一个具体的完整请求开始追踪：前端事件 → API 路由 → Controller → Service → Repository/DAO → 数据库，建立调用链认知
3. 找到与我的需求最相关的模块边界，只深入阅读这部分代码
4. 改动前先理解接口和数据模型，改动后确认上下游行为未被破坏
5. 使用 Claude、Cursor 辅助代码检索和理解，提高速度

## Q9：你在这个项目中学到了什么？

**参考回答：**

几个最有价值的收获：

1. 进入大型存量代码库的方法：先理解边界，再做最小改动
2. Agent 路由设计的思路：并非所有请求都需要完整 Agent 链路，意图分层路由可以显著提升效率
3. SSE 长连接在生产环境下的稳定性问题（Nginx 缓冲和超时是实际坑点）
4. QA 数据的工程化价值：对话产生的知识资产如何通过规范化流程回流到知识库
5. 上游框架能力与个人贡献的边界意识：在大型平台二次开发时必须清楚区分"平台提供的"和"我做的"

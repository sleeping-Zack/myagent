# 法奥机器人智能客服平台技术挑战

## 挑战 1：理解并进入大型存量代码库

### 问题

AIFlowy 是一个已有数万行代码的 Java/Vue 平台，没有全局说明文档，模块之间依赖复杂。

### 应对方式

- 从启动入口和核心路由开始，追踪一个完整请求的调用链
- 先找到与需求相关的核心模块（规划、执行、会话），定位后再做改动
- 改动前先读懂相关接口和数据模型，确保不破坏上下游
- 使用 AI 工具（Claude、Cursor）辅助代码理解和检索

### 收获

建立了进入陌生大型项目的方法论：先理解边界，再做最小改动，逐步扩展。

## 挑战 2：简单意图路由与 Agent 流程分离

### 问题

AIFlowy 原始实现将所有对话请求都导入完整 Agent 链路（规划 → 执行 → 工具调用），导致简单的寒暄、确认类消息也消耗大量 Token 和延迟，用户体验差。

### 应对方式

- 在入口处增加意图识别层：先用规则匹配快速拦截（问候、感谢等），次用 Qwen 做意图分类
- 识别为简单对话的请求直接交给对话模型，跳过规划和工具执行
- 意图识别失败或置信度低时，默认进入 Agent 流程，不影响复杂场景

### 关键取舍

误判风险：规则和模型分类都可能误判。当前设计倾向于"宁可多走 Agent"，确保不漏掉需要工具调用的复杂请求。

## 挑战 3：SSE 在 Nginx 反向代理下的稳定性

### 问题

动态规划模块通过 SSE 向前端推送执行进度，但 Nginx 代理默认会缓冲响应，导致 SSE 事件不能实时到达前端；同时默认超时设置会导致长时间任务连接被中断。

### 应对方式

- 关闭 Nginx 缓冲：`proxy_buffering off` + `X-Accel-Buffering: no`
- 增加超时配置：`proxy_read_timeout 3600s`
- 增加心跳事件，避免空闲连接被关闭

GitHub 提交中有对应的 Nginx 配置改动记录，可查阅验证。

## 挑战 4：QA 数据幂等处理

### 问题

历史会话 QA 抽取任务可能被多次触发（手动触发、定期任务），同一条会话可能被多次写入审核队列。

### 应对方式

- 以 `message_id` 作为唯一键，写入前检查是否已存在
- 已存在的记录跳过而不是更新，保证人工审核状态不被覆盖
- 抽取脚本设计为幂等操作，多次运行结果一致

## 挑战 5：个人贡献与平台能力的边界厘清

### 问题

AIFlowy 本身已有 RAG 检索、多模型接入、插件系统等能力，容易在描述项目时将平台功能混入个人贡献，造成夸大。

### 应对方式

- 明确界定：平台已有的能力（RAG、多模型、基础会话管理）属于上游框架，不作为个人成果
- 个人贡献聚焦于：Direct/Agentic 路由改造、动态规划模块、QA 数据闭环、售后工单、SSE 稳定性配置
- 面试中主动说明分工边界，用 commit 对照表区分个人工作范围

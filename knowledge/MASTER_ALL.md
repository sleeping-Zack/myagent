## RAG 检索链路

1. 将机器人领域 PDF / TXT 文档切分为 Chunk。
2. 使用 Embedding 模型生成向量并存入 Chroma。
3. Dense 召回覆盖语义相关内容。
4. BM25 补足型号、部件名和精确词匹配。
5. RRF 融合双路排名。
6. 可选 Cross-Encoder Reranker 精排。
7. 返回结构化 Evidence 与引用来源。
8. 将证据交给模型生成回答。

## 当前离线结果

当前简历记录：

- 6 份领域文档
- 326 个 Chunk
- 30 条冻结检索测试样本
- Recall@5：80.6% → 93.3%
- MRR：0.878 → 0.933
- nDCG@5：0.768 → 0.904
- Hybrid 检索 P95：385ms

这些数据属于小规模离线评测结果，用于证明方案改进方向，不等同于真实线上大规模效果。

## Harness 控制层

### AgentRunner / AgentState

统一维护 request_id、session_id、tenant_id、user_role、scene、steps、observations、tool_calls、artifacts 和 status。

### BudgetManager

支持最大步骤数、最大工具调用数、Token、成本和 Deadline 五类约束。模型或工具调用前预留预算，执行后提交实际用量，失败时释放。

### ToolPolicy

依据 tenant、role、scene、tool 和 args 返回 allow、deny、need_approval 或 need_redaction。

### HITL

敏感工具请求进入 SQLiteApprovalStore，审批后再次校验工具、参数和租户，避免从其他入口绕过控制面。

### AnswerVerifier

检查 Evidence、引用、空答案、Claim 与证据对齐和高风险结论，必要时调用 LLM Judge。

### 1. Direct / Agentic 两级路由

- 原始问答内容保留
- 人工编辑与审核
- Markdown 批量导出

支持将审核后的优质问答整理为知识库文档，形成“历史会话—问答筛选—人工审核—文档导出—知识库回流”的数据沉淀流程。

依据用户身份和所属关系进行工单数据隔离，用户端仅访问本人提交的工单，管理端负责查询、处理和更新工单状态。

支持平台在企业内网环境中进行私有化部署和统一启动，参与处理服务依赖、环境配置和长连接中断等部署问题。

## RAG 回答流程

1. 校验空问题和长度。
2. 检测 API Key、私有 IP、PEM 等敏感内容模式。
3. 将问题转换为向量。
4. 在 pgvector 中检索公开且可信的数据。
5. 在向量召回基础上叠加标题、标签和项目相关权重。
6. 判断证据是否充分。
7. 证据不足时保守拒答。
8. 先通过 SSE 返回 source 事件。
9. 加载最近 6 条对话上下文。
10. 调用 DeepSeek 流式生成。
11. 保存答案、引用、模型、Token 和延迟。
12. 支持用户反馈。

## Embedding

支持本地 BGE 模型和兼容 OpenAI Embeddings 接口的 API 模式，可使用 SiliconFlow 的 BGE 中文 Embedding 服务，降低服务器本地推理负担。

## 部署

# 9. 核心项目四：情绪分析日记与 AI 心情助手

- 仓库：https://github.com/sleeping-Zack/mood_tracker
- 性质：个人机器学习项目
- 角色：机器学习与 Django 应用开发
- 定位：传统机器学习中文文本分类与 Web 应用集成实践

## 项目背景

面向大学生日记和日常情绪记录场景，构建本地运行的中文情绪文本五分类系统。用户输入一段日记或心情文本后，系统将其识别为难过、焦虑、生气、平静或开心，并保存分析结果、生成情绪趋势图和对应提示语。

项目保留了原有 Django 页面和 SQLite 数据存储功能，同时将默认情绪识别流程改造为本地传统机器学习模型，避免网页推理依赖外部在线接口。

## 技术栈

Python、scikit-learn、pandas、jieba、TF-IDF、MultinomialNB、LogisticRegression、LinearSVC、joblib、Django、SQLite、Matplotlib。

Hugging Face 模型仅作为对照基线，K-Means 和 LSA + K-Means 仅作为无监督拓展实验，不是系统默认推理链路。

## 机器学习流程

### 数据准备与标签体系

将公开中文多情绪对话数据映射为五类情绪：

- 难过
- 焦虑
- 生气
- 平静
- 开心

对不属于目标标签体系的文本进行过滤，并针对公开数据中焦虑类样本不足的问题，补充大学生考试、答辩、项目截止时间、代码调试和实习面试等场景的焦虑文本。

完成文本去重后，使用分层抽样将数据划分为：

| 数据集 | 样本数 |
|---|---:|
| 训练集 | 3436 |
| 验证集 | 429 |
| 测试集 | 430 |

### 文本预处理

实现中文文本预处理模块，包括：

1. Unicode 字符规范化和特殊字符清理。
2. 使用 jieba 进行中文分词。
3. 加载自定义词典，改善校园和情绪相关词语的切分。
4. 使用停用词表过滤缺少分类意义的高频词。
5. 分别生成训练、验证和测试数据的分词结果。

### 特征工程

使用 `TfidfVectorizer` 将分词文本转换为稀疏向量，采用 Unigram + Bigram 特征，同时限制最大特征数量、低频词和高频词范围。

当前训练流程最终形成 3439 维文本特征，用于后续分类器训练和预测。

### 模型训练与选择

在相同数据和 TF-IDF 特征上训练并比较三种传统机器学习分类器：

- Multinomial Naive Bayes
- Logistic Regression
- LinearSVC

其中 Logistic Regression 和 LinearSVC 使用类别权重平衡，降低类别数量差异对训练结果的影响。

模型选择以验证集 Macro-F1 为主要指标，并参考 Weighted-F1、Accuracy、Macro Precision 和 Macro Recall。当前系统选择 LinearSVC 作为默认模型。

### 离线评估结果

在 430 条独立测试样本上，当前 LinearSVC 的离线结果为：

- Accuracy：91.86%
- Macro-F1：92.08%
- Weighted-F1：91.84%

同时输出五分类混淆矩阵、模型对比表和误分类样本，分析不同情绪类别之间的混淆情况。

以上指标仅代表当前数据集和划分方式下的小规模离线实验结果，不等同于开放场景下所有中文情绪文本的实际效果。

## Django 应用集成

训练完成后，使用 joblib 保存 TF-IDF 向量器和分类模型。Django 接收用户输入后，执行以下流程：

```text
文本清洗
→ jieba 分词与停用词过滤
→ TF-IDF 向量化
→ LinearSVC 分类
→ 返回五分类情绪
→ 保存至 SQLite
→ 展示历史记录与趋势图
```

模型和向量器采用进程内懒加载方式，避免每次请求重复读取模型文件。页面展示情绪类别、趋势分值、模型名称、分析时间、提示语和历史情绪变化。

## 个人贡献

## 局限

- 部分焦虑类样本为人工补充，可能存在表达模板化和数据偏差。
- 测试集规模较小，结果不能直接代表真实开放场景。
- 数据主要集中在中文对话和大学生日记场景，跨领域泛化能力有限。
- TF-IDF 和线性分类器对反讽、复杂否定和长文本上下文的理解能力有限。
- LinearSVC 的决策分数不能直接作为严格概率解释。
- 系统只用于情绪文本分类和趋势记录，不具备心理诊断能力。

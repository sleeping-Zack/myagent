# 技术面试问答知识库

本文件收录技术面试中最常被追问的问题及参考回答，聚焦 Agent、RAG 和 AI 工程方向。

---

# Agent 与 Harness

## ReAct Agent 和 Harness 有什么区别

ReAct 是模型驱动的执行机制，决定何时思考和调用工具。Harness 是请求级确定性控制层，负责预算、权限、审批、状态、验证、Trace 和 Artifact。二者分别解决"怎么执行"和"执行边界是什么"。

ReAct 负责模型的推理和工具调用，但不天然解决权限、预算、审批、证据、成本和审计。Harness 将这些非确定性执行之外的规则集中到确定性控制面。

## 为什么不能把控制逻辑都写在 Prompt 里

Prompt 不是可靠的强制边界。模型可能忽略、误解或被注入攻击影响。权限、预算和审批必须在模型之外由代码执行。

## BudgetManager 为什么需要预留

调用前预留可以保证剩余额度足够，避免并发或多轮调用在执行完成后才发现超限。成功后提交实际用量，失败则释放。

当前实现五类约束：max_steps、max_tool_calls、max_tokens、max_cost、deadline。

## HITL 如何设计

敏感工具请求先生成审批记录，返回 pending_approval。具备 operator 或 admin 权限的人批准后，系统校验租户、工具和参数，再恢复执行。

普通用户得到 pending_approval；operator / admin 批准或拒绝；审批后校验参数和租户；防止 MCP 或其他入口绕过控制面。

## MCP 会不会绕过 ToolPolicy

不能。MCP `tools/call` 必须进入同一 ToolPolicy 和 ApprovalStore，否则外部入口会绕过治理边界。

## ToolPolicy 如何判断

依据 scene、user_role、tenant_id、tool_name、args/risk，返回 allow、deny、need_approval 或 need_redaction。

## AnswerVerifier 检查什么

- 是否存在 evidence
- 是否存在引用
- 是否空答案
- 工具结果是否有效
- 是否应重试或拒答

失败时保存 verification_failure artifact。

## AgentRunner / AgentState 维护哪些字段

统一维护：request_id、session_id、tenant_id、user_role、scene、budget、steps、observations、tool_calls、artifacts、status。

# RAG

## 为什么纯 Dense 对型号和部件名召回差

Embedding 更擅长语义相似，对型号、错误码和精确词可能弱。BM25 对词面匹配更敏感，因此通过 Dense + BM25 双路召回再用 RRF 融合。

## RRF 的优点

不要求不同检索器分数同尺度，只使用排名进行融合，适合组合 Dense 和 BM25。

## 为什么需要 Reranker

第一阶段召回追求覆盖，Reranker 使用 Query—Document 交互模型提高 Top K 排序精度，但会增加延迟和成本。

## 如何构建 Retrieval Golden Set

1. 固定语料和版本
2. Dense、BM25、Hybrid 生成候选并集
3. 人工进行 0—3 级相关性标注
4. 校验标注完整性
5. 固定随机种子划分 dev/test
6. 计算 Recall、MRR、nDCG

## 为什么不能用答案关键词自动标注

会把当前检索或答案逻辑当成真值，形成自证循环，无法真实评价相关性。候选分数和答案关键词不能直接作相关性标签，否则会造成自证循环。因此将候选生成、人工判断、Golden Set 和评测拆分。

## 30 条样本够吗

只能用于快速回归和方向判断，不足以证明广泛泛化。需要扩大问题类型、增加独立 test、做置信区间和 Bad Case 分类。

## RAG 和 Fine-tuning 的区别，什么时候用哪个

核心区别是知识更新方式不同：

- **RAG**：知识存在外部数据库里，检索时动态拿进来。优点是知识可以随时更新、不用重新训练，成本低，且能给出引用来源，透明度高。缺点是检索质量影响答案质量，有检索失败的风险。
- **Fine-tuning**：把知识"烧"进模型权重里。优点是推理时不需要检索，速度快，可以调整模型行为风格。缺点是训练成本高，知识更新需要重新训练，容易遗忘。

**什么时候用 RAG**：知识库频繁更新、需要引用来源、知识量大但不需要模型掌握每条细节，比如企业文档问答、客服系统。

**什么时候用 Fine-tuning**：需要调整模型风格、格式输出要求严格，或者知识是稳定的专业领域知识且需要很高推理速度。

两者可以结合：用 Fine-tuning 调整模型的回答风格，用 RAG 补充实时知识。

## 如何处理 RAG 中的幻觉问题

幻觉主要来自两个来源：

**检索失败导致的幻觉**：检索到的内容与问题无关，但模型仍然"编"了一个答案。
- 解决：设置相似度阈值，低于阈值的结果不拿去生成；在 Prompt 中明确要求"如果提供的材料中没有相关信息，直接说不知道"。

**模型自身的幻觉**：模型忽视了检索内容，用自己训练时的"记忆"作答。
- 解决：Prompt 中强调"只能基于以下材料作答，不得使用材料之外的知识"；对答案做引用校验。

## 切块策略怎么选

切块核心原则是：一个 chunk 应该回答一类问题。

常见策略：
- **固定大小切块**：按字数或 token 数切，简单但会打断语义。适合格式规则的短文档。
- **按段落/章节切块**：按 Markdown 标题或自然段落切，语义更完整。适合有结构的文档（技术手册、FAQ）。
- **滑动窗口**：固定大小 + 一定重叠，防止关键信息被切断。适合连续文本。
- **语义切块**：用 embedding 相似度判断语义断点，精确但慢。

在有章节结构的文档中，常用按 H2 标题切块，段落超过 500 字再细分，同时在每个 chunk 前面保留标题上下文，帮助模型理解片段的位置。

## 向量检索的原理是什么

核心思路是：把文本映射到高维向量空间，语义相近的文本在这个空间里距离也近，然后用最近邻搜索找相关内容。

具体步骤：
1. 用 Embedding 模型把文本转成向量
2. 把所有文档向量存入向量数据库
3. 查询时把问题也转成向量
4. 用余弦相似度或 L2 距离找最近的 K 个向量
5. 返回对应的文档片段

**为什么用余弦相似度而不是 L2 距离**：文本向量的长度意义不大，关心的是方向（语义），余弦相似度归一化了长度影响，所以更常用。

**近似最近邻（ANN）**：真实场景数据量大，精确最近邻搜索很慢，一般用 HNSW、IVF 等 ANN 算法，用少量精度损失换速度。

# Evaluation

## Tool Recall 是什么

在需要调用特定工具的样本中，系统是否正确调用了预期工具。

## Keyword Recall 的局限

只能检查答案是否包含预期关键词，不能充分评价事实正确性和完整性，因此需要 Evidence、引用、人工评审或 LLM Judge。

## LLM-as-Judge 的风险

存在模型偏好、提示敏感、同源偏差和稳定性问题。应使用固定 rubric、独立模型、抽样人工复核和版本记录。

## 如何评估 RAG 的效果

从三个维度评估：

**检索质量**：
- Recall@K：相关文档是否在 Top-K 结果里
- MRR（Mean Reciprocal Rank）：最相关文档排在第几位
- nDCG@K：折损累计增益

**答案质量**：
- 忠实度（Faithfulness）：答案是否完全基于检索到的内容，有没有幻觉
- 答案相关性（Answer Relevancy）：答案是否真正回答了问题

**端到端**：
- 人工抽样评估，模拟真实用户提问
- 记录"无法回答"的比例

# FastAPI / SSE

## 为什么用 SSE 而不是 WebSocket

聊天主要是服务器单向持续输出，SSE 实现简单、支持 HTTP 基础设施和自动重连。需要双向实时控制时再考虑 WebSocket。

## SSE 为什么会被 Nginx 中断

代理缓冲、读取超时、心跳缺失和上游连接配置都可能导致断流，需要关闭缓冲、增加 proxy_read_timeout 并发送 heartbeat。

## FastAPI 的异步机制是怎么工作的

FastAPI 基于 ASGI，底层用 asyncio 事件循环。`async def` 路由函数在遇到 `await` 时会让出控制权，事件循环可以处理其他请求。对于 I/O 密集型操作效果显著；CPU 密集型任务需要用 `run_in_executor` 放到线程池，否则会阻塞事件循环。

# Testing

## 为什么使用 FakeBackend

大模型调用有成本、延迟和随机性。FakeBackend 可以确定性覆盖状态机、预算、审批和失败分支，隔离模型不确定性和 API 费用，使核心控制逻辑可以确定性测试。

## 如何测试 Prompt Injection

准备恶意输入和恶意检索片段，验证模型输入安全检查、检索污染扫描、工具权限和输出拒绝是否按预期工作。

# Database

## SQLite 的优点和局限

优点是零运维、便于本地演示和测试。局限是并发写、分布式部署、权限和运维能力有限。生产环境可以迁移 PostgreSQL、Redis 和对象存储。

## Chroma 的定位

适合单机作品，便于本地演示和测试。不代表分布式生产架构，大规模生产环境需要专用向量数据库或 pgvector 等方案。

## IoT 项目为什么用 Trigger 和 Procedure

课程目标是展示数据库内过程化逻辑。触发器适合插入后自动告警和入队，存储过程用于批量同步和日报。但实际大型系统也要考虑可观测性、可测试性和业务逻辑分层。

# 安全

## Prompt Injection 和检索污染的区别

Prompt Injection 是攻击者在用户输入中嵌入指令，企图操控模型行为。检索污染是攻击者在知识库或被检索的外部内容中嵌入恶意指令，通过检索链路注入模型。两者都需要在各自入口处拦截，不能只防一侧。

## Semantic Cache 的错误复用风险

如果两个语义相似但答案不同的问题命中同一缓存（如"A 型号价格"和"B 型号价格"），会返回错误答案。需要合理设置相似度阈值（当前项目设置为 0.92），并对缓存命中结果做基本校验。

# 已知局限（agentproject）

- Chroma 和 SQLite 适合单机作品，不代表分布式生产架构
- 缺少真实用户量、SLA、业务转化和线上成本
- `/chat/stream` 的当前实现更接近 SSE 事件外观，需区分完整 Token 级流式
- 标注集和 Golden Set 仍偏小
- 依赖外部模型 API
- 生产环境仍需 JWT、集中存储、队列、分布式追踪和密钥管理
